Avant de commencer

Tu te dis peut-être que publier un article sur son site, c'est simple. En théorie, oui. En pratique, ça vire vite à la corvée : copier-coller le template HTML, remplacer 15 placeholders (les emplacements à remplir, type {{TITRE}}), vérifier que rien ne casse, mettre à jour le sitemap (la carte du site lue par Google), ajouter le lien dans le parcours d'apprentissage...

Bref, 3 heures pour un article de 7 minutes de lecture. J'ai décidé de craquer là-dessus.

Mon avis en 5 secondes

Tu perds plus de temps à déployer qu'à écrire ? C'est le signe qu'il te faut un pipeline. Quelques lignes de Node suffisent — pas besoin de monter une usine.

Le concept

Le pipeline fait une chose simple : tu écris en markdown, il génère l'HTML. Comme un static site generator artisanal (en clair : un outil maison qui transforme tes textes en pages web), mais pour mon site spécifiquement, avec mon template et ma charte.

Les 3 pièces :

  1. Un dossier drafts/ où je dépose mes articles en .md
  2. Un template HTML (articles/_TEMPLATE.html) qui fixe la mise en page
  3. Un script publish.js qui fait la conversion

Quand je lance npm run publish mon-article, il lit le draft, injecte dans le template, écrit dans articles/mon-article.html, met à jour le sitemap.xml. Je relis le résultat, je commit, Vercel redéploie tout seul.

Ce que ça change concrètement

Cas 01 · Écrire avec Claude Code

Claude me fait le premier jet, je retravaille

J'ouvre Claude Code, je lui dis "écris un article sur X en suivant AGENT_BRIEF.md". Il me sort un draft markdown complet avec frontmatter (l'en-tête de méta-données : titre, date, catégorie), structure, callouts (les encadrés de mise en avant). Je relis, j'édite les passages qui sonnent faux, je valide.

Temps passé à écrire : 45 minutes (vs 2h avant). Qualité : identique, parfois meilleure parce que Claude me propose des angles auxquels je n'aurais pas pensé.

Cas 02 · Publier en une commande

Un script, trois effets

Je lance npm run publish demo-pipeline-publish. Le script :

✓ articles/demo-pipeline-publish.html généré
    ✓ sitemap.xml — nouvelle entrée ajoutée
    • parcours_etape non défini → apprendre.html non modifié

Je vérifie visuellement que tout est propre. J'ouvre le HTML en local, je scroll. Si c'est OK, je commit et je push.

Cas 03 · Déploiement Vercel auto

Git push, Vercel déploie

Vercel est branché sur mon repo GitHub (mon dépôt de code). Dès qu'un commit (un enregistrement de modif) arrive sur main (la branche principale), il détecte les fichiers modifiés et redéploie le site. Temps moyen : 30-45 secondes.

Aucune config à gérer, aucune CI custom (chaîne de tests automatiques sur mesure) à maintenir. C'est la magie des sites statiques : on publie, ça marche.

Les pièges que j'ai rencontrés

Piège 1 · Vouloir tout parser tout de suite

Mon premier réflexe a été de supporter 100% de la syntaxe markdown + 50 composants custom. Résultat : 800 lignes de script pour une utilité marginale. Je suis revenu en arrière : marked + gray-matter (deux librairies prêtes à l'emploi qui font le boulot) + quelques regex (motifs de recherche dans le texte) suffisent pour 95% des cas. Le reste, j'injecte du HTML brut dans mon markdown.

Piège 2 · Oublier les métadonnées

Au début, je publiais sans mettre à jour sitemap.xml ni la carte apprendre.html. Google finit par indexer, mais ça traîne. Le script fait ces MAJ automatiquement maintenant — un piège de moins.

Piège 3 · Le cache Vercel

Une fois, j'ai commit un article avec un typo (une coquille) dans la meta description (le résumé qu'affichent Google et les réseaux). Corrigé, pushé à nouveau, mais l'Open Graph (l'aperçu carte avec image quand tu colles un lien sur LinkedIn/X/WhatsApp) continuait à montrer la vieille version sur LinkedIn. Leçon : Vercel met un temps à invalider les previews OG. Prévoir une relecture sérieuse avant le premier push.

Ma check-list avant publish

(1) Le frontmatter est complet. (2) J'ai relu le MD à voix haute (test ton Leo). (3) Les liens internes pointent vers des articles qui existent. → Si OK aux 3, npm run publish.

Pour aller plus loin

Ce pipeline est la Phase 1 de mon plan agent IA. Ce qui arrive ensuite :

  1. Phase 2 · GitHub Actions (les automatisations de GitHub qui se déclenchent toutes seules) — un agent qui ouvre une PR (Pull Request — une proposition de modification à valider) avec un draft dès que je crée une issue (un ticket) "idée article : X". Je relis, je merge (je fusionne dans le code principal).
  2. Phase 3 · Editorial Mesh (en clair : maillage éditorial — un réseau d'agents qui se coordonnent) — plusieurs agents spécialisés (Scout, Writer, Editor) qui tournent en parallèle. Pas avant 50 articles validés.

Si tu veux voir comment évolue cette stack, réponds à la newsletter — je publierai les retours d'XP dès qu'il y a quelque chose à raconter.
